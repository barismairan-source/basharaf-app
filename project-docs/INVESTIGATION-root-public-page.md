# بررسی: صفحه‌ی روت عمومی (/) — نیاز به تصمیم کاربر

> تاریخ: ۱۴۰۵-۰۴-۰۶ (2026-06-27) · اکانت ۱ · نسخه پایه: `0.9.43`
> **وضعیت: روت دست‌نخورده ماند — منتظر تصمیم کاربر (طبق دستور خودِ کاربر).**

## هدف کاربر
وقتی کسی `basharaf.me` (روت) را باز می‌کند، یک صفحه‌ی تمیز عمومی با دکمه‌ای برای فرم استخدام ببیند، بدون هیچ نشانه‌ای از سیستم داخلی.

## وضعیت فعلی روت
`app/page.tsx` یک server component است که فوراً `redirect('/login')` می‌کند. پس هرکس روت را باز کند → به صفحه‌ی login می‌رود.

## آیا redirect روت بار-بردارِ مسیر ورود است؟ → **خیر**
بررسی کامل مسیر احراز هویت نشان داد هیچ بخشی به redirect روت وابسته نیست:
- ورود موفق → `router.replace('/dashboard')` (`(auth)/login/page.tsx:35`).
- اگر از قبل لاگین باشد → `replace('/dashboard')` (خط ۳۱).
- خروج → `router.replace('/login')` (`Sidebar.tsx:114`).
- انقضای نشست → `window.location.replace('/login')` (`store/index.ts:204`).
- middleware کاربر ناشناس روی مسیر محافظت‌شده → `/login?redirect=...`.

هیچ‌کدام از روت عبور نمی‌کنند. login مستقیماً روی `/login` کار می‌کند.

## ⚠️ اما ساختن روت عمومی یک تغییر ظریفِ auth لازم دارد
`SessionSync` در root layout روی **همه‌ی** صفحات اجرا می‌شود و `bootstrap()` یک `GET /api/auth/me` می‌زند. برای کاربر ناشناس ۴۰۱ می‌گیرد و **به `/login` ریدایرکت می‌کند، مگر** مسیر فعلی در این allowlist باشد (`store/index.ts:199`):
```
['/login', '/signup', '/forgot', '/apply', '/m', '/order']
```
**روت `/` در این لیست نیست.** پس اگر روت را به یک صفحه‌ی محتوادار تبدیل کنیم، بازدیدکننده‌ی ناشناس فوراً به `/login` پرت می‌شود.

→ ساختن روت عمومی **دو تغییر** می‌خواهد:
1. صفحه‌ی جدید `app/page.tsx` (به‌جای redirect).
2. افزودن `'/'` به allowlist در `store/index.ts:199` (تغییر منطق auth-adjacent).

## چرا نزدم
دستور صریح کاربر: «اگر به تصمیمی رسیدی که حتماً مسیر ورود را عوض می‌کند (مثلاً روت دیگر به لاگین نرود)، این یک مورد را نزن، در HANDOFF زیر نیاز به تصمیم کاربر یادداشت کن.»
ساختن روت عمومی دقیقاً یعنی «روت دیگر به لاگین نرود» + لمس allowlist نشست. پس flag شد.

## پیشنهاد من (آماده‌ی اجرا با یک تأیید)
**امن و کم‌ریسک:** روت را عمومی کن با این دو تغییر + یک لینک کوچک «ورود کارکنان» → `/login` تا مسیر کارکنان حفظ شود:
1. `app/page.tsx`: صفحه‌ی تمیز — برند «با شرف»، دکمه‌ی بزرگ «درخواست همکاری» → `/apply`، و یک لینک کوچک گوشه «ورود کارکنان» → `/login`. (server component، بدون نشت سیستم.)
2. `store/index.ts:199`: افزودن `'/'` به allowlist تا ناشناس روی روت نماند bounce نشود.
3. عنوان روت از قبل با بخش ۳ تمیز است («با شرف»).

**اثر تنها تغییر رفتار:** تایپ دامنه‌ی خام دیگر مستقیم به login نمی‌رود؛ کارکنان از لینک «ورود کارکنان» یا بوکمارک `/login` استفاده می‌کنند. login هیچ خرابی ندارد.

**گزینه‌ی محافظه‌کارانه‌تر (اگر نمی‌خواهی allowlist لمس شود):** روت را همان redirect نگه‌دار، و صفحه‌ی عمومی را روی یک مسیر جدا (مثل `/careers`) بساز. ولی این هدف «روت تمیز» را برآورده نمی‌کند.

→ **صبح بگو کدام را اجرا کنم.**
