# تصمیم: ستون «کانال فروش» (`channel`) روی جدول `transactions`

> **وضعیت: فقط مستندسازی (فاز ۸، نسخه 0.9.5-operations) — هیچ کدی پیاده‌سازی/migration نشده.**
> این سند فقط تصمیم و طرح را ثبت می‌کند تا در یک فاز جدا (بعد از تأیید سوالات باز بخش ۴) ساخته شود.

---

## ۱. مسئله / نیاز

تراکنش‌های `income` (فروش) در حال حاضر این اطلاعات را دارند:
`type`, `categoryId`/`categoryName`, `amount`, `payee`, `branchId`, `method` (روش پرداخت: نقد/کارت/پوز/حواله/چک), `saleMeta` (jsonb — فقط برای backflushing منو).

هیچ فیلدی نشان نمی‌دهد فروش از چه **کانالی** انجام شده:
- حضوری/سالن
- بیرون‌بر (تیک‌اوی)
- ارسال توسط رستوران (پیک خودش)
- اپلیکیشن‌های آنلاین (اسنپ‌فود، فودرو، ...)

بدون این فیلد، گزارش‌گیری از سهم هر کانال در فروش ممکن نیست.

---

## ۲. تصمیم

افزودن ستون جدید **`channel` (text, nullable)** به جدول `transactions`.

### چرا ستون مستقل، نه استفاده از `sale_meta` موجود؟
`sale_meta` (jsonb) مصرف مشخص و typed دارد: فقط برای backflushing منو
(`lines`, `deductedAt`, `cogsTransactionId`) و توسط کد انبار/حسابداری
(`app/api/transactions/[id]/approve/route.ts`, `app/api/transactions/[id]/route.ts`)
خوانده/نوشته می‌شود. اضافه کردن یک فیلد گزارشی عمومی به این jsonb عملیاتی باعث
ابهام نوع و ریسک تعارض می‌شود. ستون مستقل برای فیلتر/گروه‌بندی در گزارش‌ها
(SQL مستقیم) هم ساده‌تر است.

### چرا `text` nullable و نه `pgEnum` جدید؟
طبق قرارداد فعلی schema (بخش ۲ HANDOFF.md): «نوع/وضعیت → `text` با check، نه
pgEnum جدید». `nullable` چون تراکنش‌های قدیمی و انواع `expense`/`transfer`
معمولاً کانال ندارند.

### مقادیر پیشنهادی (برای تأیید نهایی کاربر در فاز پیاده‌سازی)
| مقدار | معنی |
|---|---|
| `dine_in` | حضوری / سالن |
| `takeaway` | بیرون‌بر |
| `delivery_own` | ارسال توسط رستوران (پیک خودش) |
| `delivery_app` | اپلیکیشن‌های آنلاین (اسنپ‌فود/فودرو/...) |

تفکیک ریزتر اپ‌های آنلاین (`app_snapp`, `app_foodfellow`, ...) ممکن است enum را
شلوغ کند — تصمیم نهایی با کاربر.

---

## ۳. اثر روی کد (در فاز پیاده‌سازی آینده)

- **Migration** (idempotent، مثل سایر `db-*-migration.sql`):
  `ALTER TABLE transactions ADD COLUMN IF NOT EXISTS channel text;`
- **`lib/db/schema.ts`**: افزودن `channel: text('channel')` به `transactions`.
- **فرم ثبت تراکنش** (`app/(app)/transactions/new/page.tsx`): فیلد select
  «کانال فروش» — احتمالاً فقط وقتی `type === 'income'`.
- **API** (`app/api/transactions/route.ts`, `[id]/route.ts`): پذیرش/اعتبارسنجی
  اختیاری `channel` در body schema (zod).
- **گزارش‌ها** (`/reports`, dashboard `BreakdownCard`): بُعد جدید گروه‌بندی فروش
  بر اساس کانال.
- **فیلتر صفحه تراکنش‌ها**: فیلتر بر اساس کانال.

---

## ۴. سوالات باز (باید قبل از پیاده‌سازی با کاربر روشن شود)

1. کانال فقط برای `income` معنا دارد یا برای `expense` هم (مثلاً هزینه‌ی
   کمیسیون اپ مرتبط با `delivery_app`)؟
2. لیست دقیق مقادیر enum — اپ‌های آنلاین تفکیک شوند یا یک مقدار کلی
   `delivery_app`؟
3. آیا انتخاب کانال برای تراکنش‌های `income` جدید **اجباری** شود یا اختیاری؟
4. در گزارش breakdown فعلی (`BreakdownCard`) بُعد کانال هم نمایش داده شود یا
   فقط در Excel export؟
5. تراکنش‌های قدیمی (`channel = null`) backfill شوند با یک مقدار پیش‌فرض
   (مثلاً `dine_in` یا «نامشخص») یا `null` بمانند؟

---

## ۵. خلاصه

این سند فقط مستند تصمیم است. پیاده‌سازی (migration + schema + UI + API + گزارش)
در یک فاز جدا، بعد از پاسخ کاربر به سوالات بخش ۴، انجام می‌شود.
