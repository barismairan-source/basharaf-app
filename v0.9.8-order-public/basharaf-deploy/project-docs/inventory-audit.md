### جریان ۱: تأیید برگه خرید (kind='in')

**مسیر کد:** `approve/route.ts` → `approveVoucherTx` → `receiveConfirmed` → `postPurchaseToAccounting`

#### محاسبه‌ی WAC

```ts
// inventoryHelpers.ts — receiveConfirmed
const oldVal = oldQty * oldAvg;
const newQty = oldQty + qtyBase;
const newAvg = newQty > 0 ? (oldVal + totalCost) / newQty : 0;
```

فرمول: `(موجودی_قبلی × میانگین_قبلی + مقدار_جدید × قیمت_جدید) ÷ (موجودی_قبلی + مقدار_جدید)`

این فرمول استاندارد میانگین موزون است. ✅

**اولین خرید (qtyBase قبلی = 0):**
- `oldVal = 0`، `newAvg = totalCost / qtyBase` → درست ✅

**دو برگه برای یک ماده:** دو بار `receiveConfirmed` صدا زده می‌شود. WAC بعد از خط اول درست محاسبه، و سپس خط دوم روی آن اعمال می‌شود. از نظر ریاضی معادل ادغام دو خط در یکی است. ✅

**قیمت نهایی صفر:**
اسکیمای تأیید: `z.number().finite().min(0)` — حداقل صفر. اگر تأییدکننده قیمت صفر وارد کند، `receiveConfirmed(qty, 0)` اجرا می‌شود:
- `newAvg = (oldVal + 0) / (oldQty + qty) < oldAvg` — میانگین کاهش می‌یابد
- این باگ نیست (کاربر عمداً صفر وارد کرده)، ولی UI هیچ هشداری نمی‌دهد.

#### اثر حسابداری خرید

`postPurchaseToAccounting` ← اولین حساب فعال شعبه را انتخاب می‌کند:

```ts
const [a] = await dbTx.select().from(schema.accounts)
  .where(and(eq(schema.accounts.branchId, voucher.branchId), eq(schema.accounts.isActive, true)))
  .limit(1);
```

**مشکل:** انتخاب حساب کاملاً خودکار است — اولین حساب فعال شعبه انتخاب می‌شود، بدون اینکه مسئول انبار بتواند مشخص کند هزینه به کدام صندوق/بانک خورده است. در یک رستوران با چند صندوق (نقد + POS + بانک)، همه‌ی خریدها از یک حساب کسر می‌شوند. 🟡

**اگر حساب وجود نداشته باشد:**
```ts
const accountId = account?.id ?? null;
// ...
if (accountId) await applyBalance(dbTx, coreTx);
```
تراکنش هزینه ساخته می‌شود ولی هیچ حسابی کسر نمی‌شود. مدیر در گزارش هزینه را می‌بیند ولی موجودی هیچ صندوقی تغییر نکرده. 🟡

#### بازمحاسبه‌ی خودکار نیمه‌آماده‌ها

```ts
const branchFilter = current.branchId ? eq(schema.invItems.branchId, current.branchId) : undefined;
const allItems = await dbTx.select().from(schema.invItems).where(branchFilter);
const { changed } = computeAutoRecost({ items: recostInput });
```

- `computeAutoRecost` زنجیره‌ی `resolvePrepCostChain` را برای همه‌ی نیمه‌آماده‌های شعبه اجرا می‌کند.
- شناسایی حلقه (A→B→A) پیاده‌سازی شده و خطا نمی‌دهد.
- فقط مقادیر واقعاً تغییریافته persist می‌شوند (مقایسه با `1e-6` دقت). ✅

**نتیجه‌ی جریان ۱:**
| مورد | وضعیت |
|---|---|
| فرمول WAC | 🟢 درست |
| اولین خرید | 🟢 درست |
| بازمحاسبه‌ی نیمه‌آماده | 🟢 درست |
| انتخاب حساب خودکار | 🟡 ریسک‌دار — بدون کنترل کاربر |
| اگر حسابی نباشد | 🟡 ریسک‌دار — هزینه ثبت، صندوق کسر نشده |

### جریان ۲: بکفلاش فروش منو (Menu Sale Backflushing)

**مسیر کد:** `transactions/[id]/approve/route.ts` → `applyMenuSaleDeduction` → `issueConfirmed`

#### انفجار رسپی با احتساب افت

```ts
const yieldUsed = rl.overridePct != null ? n(rl.overridePct) : (n(item.yieldPct) || 100);
const factor = yieldUsed > 0 ? 100 / yieldUsed : 1;
const qtyPerSale = (n(rl.qtyBase) / portions) * saleLine.qty * factor;
```

مثال: رسپی پیتزا، خط ۱: `qtyBase=500g` مرغ، `portions=10`، `yieldPct=80%`:
- مرغ لازم برای ۳ پرس: `(500/10) × 3 × (100/80) = 187.5g` (خام، قبل از تمیزکاری)
- درست: ۱۸۷.۵ گرم خام → ۱۵۰ گرم پاک (۸۰٪) → ۵۰g × ۳ پرس = ۱۵۰g ✅

#### وقتی موجودی کافی نیست

`issueConfirmed` یک **clamp** دارد:
```ts
const q = Math.min(qtyBase, have);  // هرگز منفی نمی‌شود
const cost = q * avg;
```

موجودی هرگز منفی نمی‌شود. ✅ ولی:

```ts
// menuSaleDeduction.ts:104
if (qtyPerSale > available + 1e-6) {
  warnings.push(`موجودی «${item.name}» کافی نیست — ...`);
}
const cost = await issueConfirmed(dbTx, item.id, qtyPerSale);
```

**۱. فروش ادامه می‌یابد (بلاک نمی‌شود):** حتی اگر موجودی کافی نباشد، تراکنش تأیید می‌شود و فقط مقدار موجود کسر می‌شود. هشدار فقط در جدول `audit_logs` ثبت می‌شود — **هیچ toast یا اعلانی به مدیر نمی‌رسد.** 🔴

**۲. COGS ناقص:** اگر برای ۵ پرس ۵۰۰g مرغ لازم باشد ولی فقط ۲۰۰g موجود باشد:
- کسر: ۲۰۰g (نه ۵۰۰g)
- COGS: `200g × WAC` (نه `500g × WAC`)
- سود ناخالص اعلام‌شده بالاتر از واقعیت خواهد بود. 🔴

**۳. دفترداری COGS وقتی موجودی کافی باشد:**
تراکنش COGS با `accountId=null` ساخته می‌شود — فقط یک ردیف دفترداری، بدون اثر بر صندوق. ✅

**نتیجه‌ی جریان ۲:**
| مورد | وضعیت |
|---|---|
| انفجار رسپی با افت | 🟢 درست |
| موجودی منفی | 🟢 بلاک (clamp) |
| موجودی ناکافی — اطلاع‌رسانی | 🔴 فقط در audit log، مدیر نمی‌بیند |
| COGS وقتی موجودی کم است | 🔴 دست‌کم محاسبه می‌شود |
| COGS دفترداری وقتی موجودی کافی | 🟢 درست (accountId=null) |

### جریان ۴: تولید نیمه‌آماده (kind='produce')

**مسیر کد:** `produce/route.ts` → `produceConfirmed` → `issueConfirmed` + `receiveConfirmed`

#### انتقال بهای تمام‌شده

```ts
// inventoryHelpers.ts — produceConfirmed
const recipe = prep.prepRecipe as Array<{ itemId: string; qtyBase: number }>;
let totalCost = 0;
for (const r of recipe) {
  totalCost += await issueConfirmed(tx, r.itemId, r.qtyBase * b);
}
const producedBase = n(prep.batchYieldBase) * b;
await receiveConfirmed(tx, prepItemId, producedBase, totalCost);
```

بهای تمام‌شده مواد خام مصرف‌شده (به قیمت WAC لحظه‌ی تولید) به نیمه‌آماده منتقل می‌شود. ✅

#### 🔴 باگ: افت (yieldPct) در مواد خام اعمال نمی‌شود

مقایسه‌ی فروش منو با تولید:

| | فروش منو (`menuSaleDeduction.ts`) | تولید (`produceConfirmed`) |
|---|---|---|
| فرمول کسر | `qtyBase × (100/yieldPct)` | `qtyBase` (بدون ضریب) |
| افت مواد خام | ✅ اعمال می‌شود | ❌ اعمال **نمی‌شود** |

مستندات `costing.ts` صریحاً می‌گوید:
> `qtyBase` = مقدار **خالص** لازم به واحد پایه
> بهای خط = qtyBase × avgCostPerBase **÷ (yield/100)**

پس `qtyBase` در `prepRecipe` مقدار **خالص** (پس از افت) است. برای کسر از انبار باید مقدار **ناخالص** (قبل از افت) استفاده شود:
```
مقدار ناخالص = qtyBase / (yieldPct/100) = qtyBase × (100/yieldPct)
```

**اثر این باگ:**
- اگر `yieldPct=80%` برای مرغ → تولید ۱۰ بچ با `qtyBase=1kg` مرغ:
  - درست: `1000g × (100/80) × 10 = 12,500g` کسر
  - فعلی: `1000g × 10 = 10,000g` کسر
  - خطا: **۲,۵۰۰g مرغ کمتر از واقعیت از انبار کسر می‌شود**
- COGS نیمه‌آماده کمتر از واقعیت است
- موجودی مواد خام در سیستم بالاتر از واقعیت باقی می‌ماند

**نتیجه‌ی جریان ۴:**
| مورد | وضعیت |
|---|---|
| انتقال بهای تمام‌شده | 🟢 درست (WAC صحیح) |
| اعمال افت مواد خام در تولید | 🔴 **باگ: افت اعمال نمی‌شود** |
| ناسازگاری با فروش منو | 🔴 بکفلاش افت‌دار اما تولید افت‌ندار |

### جریان ۵: انبارگردانی (Stocktake)

**مسیر کد A (مستقیم):** `stocktake/route.ts` → `stocktakeConfirmed` + درج `invStockTx`
**مسیر کد B (از طریق برگه):** `approve/route.ts` (kind='stocktake') → `approveVoucherTx` → `stocktakeConfirmed`

#### تعدیل موجودی

```ts
// inventoryHelpers.ts — stocktakeConfirmed
const diff = countedBase - sysQty;
if (diff > 0) {
  await receiveConfirmed(tx, itemId, diff, diff * avg); // مازاد: با میانگین فعلی وارد
} else {
  await issueConfirmed(tx, itemId, -diff);              // کسری: با WAC خارج
}
await tx.update(schema.invItems).set({ qtyPhysical: String(countedBase) });
```

مازاد با میانگین فعلی وارد می‌شود (WAC تغییر نمی‌کند). ✅
کسری با WAC خارج می‌شود. ✅

#### 🔴 باگ: برگه‌ی انبارگردانی — لاگ invStockTx ثبت نمی‌شود

در مسیر B، approve route صریحاً لاگ را نادیده می‌گیرد:

```ts
// approve/route.ts:69
for (const l of lines) {
  if (kind === 'stocktake') continue; // ← این خط را می‌پرد!
  // ...
  await dbTx.insert(schema.invStockTx).values({ ... });
}
```

کامنت کد می‌گوید "لاگ حرکتش داخل خودش است" — اما نه `approveVoucherTx` و نه `stocktakeConfirmed` لاگی می‌نویسند. در مسیر A (API مستقیم) لاگ نوشته می‌شود. در مسیر B (از طریق برگه) لاگ **اصلاً نوشته نمی‌شود**.

**اثر:** تمام انبارگردانی‌هایی که از طریق برگه‌ی انبار ثبت و تأیید شده‌اند، در گزارش حرکت موجودی (`inv_stock_tx`) نامرئی‌اند.

#### 🟡 ریسک: مغایرت انبارگردانی در حسابداری ثبت نمی‌شود

ضایعات → `postWasteToAccounting` → تراکنش هزینه (برای P&L)
انبارگردانی (کسری/مازاد) → **هیچ تراکنش مالی ساخته نمی‌شود**

اگر انبارگردانی نشان دهد ۵ کیلو گوشت کم است (مثلاً ارزش ۵,۰۰۰,۰۰۰ تومان):
- موجودی انبار کاهش می‌یابد ✅
- P&L هیچ هزینه‌ای ثبت نمی‌کند ❌
- این کسری در گزارش مالی دیده نمی‌شود

**نتیجه‌ی جریان ۵:**
| مورد | وضعیت |
|---|---|
| تعدیل موجودی قطعی | 🟢 درست |
| WAC هنگام مازاد | 🟢 درست (تغییر نمی‌کند) |
| لاگ invStockTx (مسیر مستقیم) | 🟢 درست |
| لاگ invStockTx (مسیر برگه) | 🔴 **باگ: هرگز نوشته نمی‌شود** |
| ثبت مالی مغایرت | 🟡 ریسک: در حسابداری نمی‌آید |

#### حذف تراکنش approved (مقایسه)

برخلاف برگه انبار، تراکنش مالی approved قابل حذف با rollback کامل است:
```
reverseBalance → reverseContactBalance → reverseSaleDeduction → voidCogsTransaction → delete
```
این rollback اتمیک و کامل است. ✅

### ۲. ماشین‌حساب تعداد پرس — 🧮 Portion Calculator

**تعریف:** «اگه ۵ کیلو مرغ داریم، چند پرس جوجه‌کباب می‌توانیم بدهیم؟» و برعکس: «برای ۸۰ پرس، چقدر مواد لازم است؟»

**وضعیت فعلی:**
- ✅ داریم: `qtyBase` در خطوط رسپی، `qtyBase` موجودی در `invItems.qtyBase`
- ✅ داریم: `yieldPct` برای محاسبه‌ی افت
- ❌ نداریم: endpoint و UI محاسبه

**منطق محاسبه:**
```
پرس ممکن از ماده X = (موجودی × yieldPct/100) / (qtyBase_per_portion)
پرس ممکن کل = min(پرس ممکن از هر ماده)
```

**تلاش:** `S` — یک تابع utility + یک endpoint جدید + UI نمایش در صفحه رسپی

**ارزش برای آشپز:** بسیار بالا — هر روز صبح می‌تواند ببیند چند پرس از هر غذا دارد.

### ۳. هشدار موجودی کم بر اساس نرخ مصرف — ⚠️ Smart Low-Stock Alert

**تعریف:** نه فقط «موجودی زیر حداقل»، بلکه «با این نرخ مصرف، در ۳ روز آینده تمام می‌شود.»

**وضعیت فعلی:**
- ✅ داریم: `minBase` در `invItems` (آستانه‌ی هشدار دستی)
- ✅ داریم: `inv_daily_sales` با تاریخچه‌ی مصرف روزانه
- ❌ نداریم: endpoint که نرخ مصرف ۷ روز گذشته را محاسبه و روزهای باقیمانده را تخمین بزند

**منطق محاسبه:**
```
نرخ مصرف = sum(COGS ماده X در ۷ روز) / 7
روزهای باقیمانده = موجودی فعلی / نرخ مصرف
هشدار اگر: روزهای باقیمانده < N (قابل تنظیم)
```

**تلاش:** `M` — نیاز به query ترکیبی `inv_stock_tx` × `inv_daily_sales` × `invItems` + UI هشدار

**ارزش برای آشپز:** بالا — جلوی تمام‌شدن ناگهانی مواد اولیه را می‌گیرد.

### ۴. هشدار انقضای مواد — 📅 Expiry Warnings

**تعریف:** برای مواد اولیه‌ای که تاریخ انقضا دارند، قبل از انقضا هشدار نمایش داده شود (مثلاً ۳ روز مانده به انقضا).

**وضعیت فعلی:**

**چالش:** از آنجا که موجودی به صورت `qtyBase` یکجا نگهداری می‌شود (نه FIFO جداگانه)، باید جمع `invStockTx` با آخرین `expiryDate` معتبر برای هر ماده محاسبه شود.

### ۵. برنامه‌ریزی تولید نیمه‌آماده — 🗓️ Prep Planning

**تعریف:** «امروز چقدر نیمه‌آماده بسازیم؟» — بر اساس میانگین فروش روزهای مشابه (مثلاً شنبه‌های ماه گذشته).

**وضعیت فعلی:**
- ✅ داریم: `inv_daily_sales` با `jalaliDate`, `lines` (recipeId + qty + cogs)
- ❌ نداریم: منطق شناسایی روز هفته از تاریخ جلالی
- ❌ نداریم: الگوریتم میانگین‌گیری روزهای مشابه
- ❌ نداریم: تبدیل «فروش X پرس» به «نیاز به Y گرم نیمه‌آماده»

**منطق محاسبه:**
```
روزهای مشابه = آخر ۴ تکرار همین روز هفته در inv_daily_sales
فروش پیش‌بینی‌شده = میانگین فروش هر آیتم در آن روزها
نیمه‌آماده لازم = فروش پیش‌بینی × qty_per_portion از رسپی
مقدار تولید = نیمه‌آماده لازم - موجودی فعلی نیمه‌آماده
```

**تلاش:** `L` — پیچیدگی در کار با تاریخ جلالی + API پیچیده + UI با درخت رسپی

**ارزش برای آشپز:** بسیار بالا برای آشپزخانه‌ی شلوغ، ولی تنها وقتی داده‌ی تاریخچه کافی باشد (حداقل ۲ ماه).
