### 🔴 بحرانی — تداخل نامی: «صندوق‌ها» دو چیز است

در گروه «عملیات اصلی» دو لینک کنار هم هستند:

| لینک | مسیر | چه چیز را نشان می‌دهد |
|---|---|---|
| صندوق‌ها و فروش | `/transactions` | تراکنش‌های درآمد/هزینه/انتقال |
| صندوق‌ها | `/accounts` | موجودی صندوق نقد/بانک/POS |

یک مدیر رستوران تازه‌کار هرگز نمی‌فهمد تفاوت «صندوق‌ها و فروش» با «صندوق‌ها» چیست.
در زبان روزمره رستوران، «صندوق» یعنی جایی که پول است — هر دو مفهوم یک اسم دارند.

**پیشنهاد اصلاح:**
- `/transactions` → «تراکنش‌ها» یا «ورود و خروج پول»
- `/accounts` → «کیف‌ها و حساب‌ها» یا «صندوق نقد و بانک»

### 🟡 باید اصلاح شود — «روابط و منابع» ۷ آیتم دارد و ناهمگون است

گروه «روابط و منابع» ترکیبی از سه دسته‌ی کاملاً جدا است:

**دسته‌ی مشتری:**
- امور مشتریان (CRM + وفاداری)
- رزرو میز
- کوپن‌ها

**دسته‌ی مالی:**
- **طرف‌حساب‌ها (contacts)** — این یک «دفتر بدهکاران و بستانکاران» است. در هر ERP رستورانی (Toast، Lightspeed) این زیر مجموعه‌ی حسابداری است، نه «روابط». یک مدیر وقتی می‌خواهد ببیند آشپز فلانی چقدر پول در حساب دارد، اول در «روابط» نگاه نمی‌کند.

**دسته‌ی HR:**
- پرسنل
- حقوق و دستمزد
- استخدام

**پیشنهاد اصلاح گروه‌بندی:**

```
عملیات روزانه:   داشبورد | تراکنش‌ها | کیف‌ها و حساب‌ها | طرف‌حساب‌ها
رستوران و منو:   انبار و آشپزخانه | مدیریت منو | رزرو میز
مشتریان:         امور مشتریان | کوپن‌ها
پرسنل:           پرسنل | حقوق و دستمزد | استخدام
گزارش‌ها:        گزارش مالی
تنظیمات:         (پین‌شده)
```

### 🔴 بحرانی — صندوق با مانده‌ی غیرصفر قابل «حذف» است

**فایل:** `app/api/accounts/[id]/route.ts:30`

```ts
// کد فعلی — هیچ چکی روی مانده نیست
await db.update(schema.accounts)
  .set({ isActive: false, updatedAt: new Date() })
  .where(eq(schema.accounts.id, params.id));
```

یک صندوق نقد با مانده‌ی ۵۰,۰۰۰,۰۰۰ تومان را می‌توان «حذف» کرد. مانده از دید UI ناپدید می‌شود، ولی در دیتابیس هست. هیچ هشدار، confirm، یا بلوک وجود ندارد.

**اثر واقعی برای رستوران:** مدیر صندوق اشتباهی را غیرفعال می‌کند، بعد تعجب می‌کند چرا گزارش مالی با واقعیت نمی‌خواند. اگر بعداً صندوق دوباره فعال شود، تراکنش‌های آن (که `account_id` را با `set null` نگه نمی‌دارند — بلکه باقی می‌مانند) درست می‌شوند. ولی این برای کاربر کاملاً ناشفاف است.

**اصلاح لازم:** قبل از soft-delete، چک کنید `balance != 0`. اگر غیرصفر بود، خطای ۴۰۹ با پیام روشن بدهید.

### 🔴 بحرانی — طرف‌حساب با بدهی/مطالبه قابل «حذف» است

**فایل:** `app/api/contacts/[id]/route.ts:30`

```ts
// کد فعلی — هیچ چکی روی مانده نیست
await db.update(schema.contacts).set({ isActive: false, updatedAt: new Date() })
```

اگر یک تأمین‌کننده (supplier) ۲۰,۰۰۰,۰۰۰ تومان از رستوران طلب داشته باشد و کاربر اشتباهاً آن را «حذف» کند، بدهی از دیدِ UI ناپدید می‌شود. از نظر حسابداری این یعنی تعهد مالی موجود از کتاب‌ها پاک شده.

**اصلاح لازم:** چک `balance != 0` قبل از soft-delete، با پیام: «این طرف‌حساب X تومان [بدهکار/بستانکار] است. ابتدا حساب را تسویه کنید.»

### 🔴 بحرانی — کوپن‌های حذف‌شده در GET برمی‌گردند

**فایل:** `app/api/coupons/route.ts:45`

```ts
// GET — بدون فیلتر isActive
const rows = await db.select().from(schema.coupons).orderBy(desc(schema.coupons.createdAt));
```

DELETE کوپن فقط `isActive=false` می‌کند، ولی GET همه‌ی رکوردها را برمی‌گرداند. کوپن «حذف‌شده» بعد از refresh برمی‌گردد — دقیقاً همان باگی که در قلم انبار بود و امروز رفع شد.

**اصلاح لازم:** `eq(schema.coupons.isActive, true)` به WHERE اضافه شود.

### 🔴 بحرانی — حذف کاربر با تراکنش: crash پنهان

**فایل:** `app/api/users/[id]/route.ts:92`

```ts
const result = await db
  .delete(schema.users)
  .where(eq(schema.users.id, params.id))
  .returning({ id: schema.users.id });
```

FK در schema:
```ts
// transactions.ts
createdBy: uuid('created_by').notNull().references(() => users.id, { onDelete: 'restrict' })
```

اگر کاربر حتی یک تراکنش ثبت کرده باشد (که اکثر کاربران دارند)، DB خطای `foreign key violation` پرتاب می‌کند. `handleError` این را به عنوان خطای عمومی بر می‌گرداند — پیام نامفهومی نظیر «خطا» به کاربر می‌رسد. هیچ پیام مناسبی مثل «این کاربر ۴۵ تراکنش ثبت کرده و قابل حذف نیست» وجود ندارد.

همچنین: `payroll_events.createdBy` نیز FK به users دارد (`set null` است پس مشکل ندارد)، اما `transactions.createdBy` با `restrict` بلوک می‌کند.

**اصلاح لازم:**
۱. قبل از حذف، تعداد تراکنش‌های ثبت‌شده توسط کاربر را چک کنید.
۲. اگر > 0 بود، خطای ۴۰۹ با پیام مناسب دهید یا soft-delete را پیشنهاد کنید.

### 🟡 باید اصلاح شود — هیچ راهی برای «بازگرداندن» موجودیت soft-delete‌شده وجود ندارد

موجودیت‌های زیر با `isActive=false` «حذف» می‌شوند:
- صندوق‌ها، طرف‌حساب‌ها، قلم انبار، دستور پخت، کوپن، مشتریان، میزها، پرسنل

**ولی در هیچ‌کجای UI نمی‌توان:**
- لیست موجودیت‌های غیرفعال را دید
- یک موجودیت را بازگرداند (restore)

اگر کاربر اشتباهاً صندوقی را حذف کند، تنها راه بازگردانی آن، دسترسی مستقیم به دیتابیس است.

**پیشنهاد:** یک toggle «نمایش آرشیوشده‌ها» در هر صفحه + دکمه «بازگرداندن» کافی است. این یک UX issue است، نه فقط تکنیکی.

### 🟡 باید اصلاح شود — ناهماهنگی نوع حذف در موجودیت‌های مشابه

هیچ قاعده‌ی مشخصی برای اینکه چه چیزی soft و چه چیزی hard است وجود ندارد:

| نوع | Soft | Hard | Void |
|---|---|---|---|
| دارد تاریخچه‌ی مالی | صندوق، طرف‌حساب | شعبه، کاربر، دسته‌بندی | — |
| ندارد تاریخچه‌ی مالی | قلم انبار، مشتری، میز، کوپن | رزرو، درخواست استخدام، آیتم منو | رویداد حقوق |

**قاعده‌ی منطقی که وجود ندارد:** هر موجودیتی که تراکنش مالی به آن وابسته است باید soft-delete باشد. اما شعبه و کاربر (که هر دو `restrict` FK دارند) hard delete هستند — یعنی عملاً حذف‌ناپذیرند ولی UI دکمه‌ی حذف دارد.

## بخش ۳ — مدل حسابداری: تک‌ستونه در مقابل دوطرفه

### وضعیت فعلی

سیستم **single-entry** است:
- هر تراکنش یک رکورد با `type` (income/expense/transfer) و `amount`.
- موجودی صندوق‌ها با `applyBalance` / `reverseBalance` مستقیم تغییر می‌کند.
- جدول `journal_vouchers` وجود دارد ولی **فقط برای حقوق** — عمومی نیست.

---

### 🔴 مشکلات عملی برای یک رستوران واقعی

**۱. ترازنامه (Balance Sheet) غیرممکن است**
ترازنامه نیاز به تفکیک دارایی/بدهی/حقوق صاحبان سهام دارد. در سیستم فعلی هیچ نوع دارایی‌بندی حسابداری وجود ندارد. `accounts` فقط صندوق نقد/بانک است، نه «دارایی» به مفهوم حسابداری.

**۲. صورت سود و زیان (P&L) ناقص است**
گزارش فعلی (`/reports`) فقط مجموع درآمد و هزینه نشان می‌دهد. یک P&L استاندارد نیاز دارد به:
- COGS (بهای تمام‌شده فروش) — در سیستم کنونی این از backflushing محاسبه می‌شود ولی در گزارش‌ها یکجا نمی‌آید
- سود ناخالص (Gross Profit)
- هزینه‌های عملیاتی (حقوق، اجاره، ...)
- EBITDA

**۳. حساب‌های پرداختنی/دریافتنی (AP/AR) ابتدایی است**
`contacts.balance` یک عدد یگانه است — فقط «طلب داریم» یا «بدهکاریم». هیچ aging report (گزارش سن‌بندی بدهی‌ها) وجود ندارد. برای یک رستوران با ۱۵+ تأمین‌کننده، این کافی نیست.

**۴. تسویه بانکی (Bank Reconciliation) ممکن نیست**
برای مقایسه موجودی کتاب با صورت‌حساب بانک، به journal entries با reference number نیاز است. این وجود ندارد.

**۵. VAT گزارش‌پذیر نیست**
`vat_amount` در تراکنش‌ها ثبت می‌شود ولی هیچ گزارش VAT/مالیاتی وجود ندارد. برای اداره مالیات باید query مستقیم دیتابیس بنویسند.

---

### تخمین مهاجرت به double-entry

| مرحله | کار | تخمین |
|---|---|---|
| طراحی chart of accounts | تعریف سرفصل‌های حسابداری (صندوق، بانک، درآمد، هزینه، COGS، بدهی‌ها...) | ۳–۵ روز |
| جدول `journal_lines` | هر تراکنش → ۲+ خط بدهکار/بستانکار | ۲–۳ روز |
| ماژول posting | تبدیل income/expense/transfer به journal entries | ۵–۷ روز |
| مهاجرت داده‌های قدیمی | تبدیل تراکنش‌های موجود | ۳–۵ روز |
| گزارش‌های جدید | Trial balance، P&L کامل، ترازنامه | ۷–۱۰ روز |
| **جمع** | | **~۲۰–۳۰ روز کاری** |

**توصیه:** فعلاً مهاجرت توجیه ندارد. سیستم برای یک رستوران کوچک تا متوسط کافی است. اما اگر بخواهند به حسابدار یا اداره مالیات گزارش بدهند، باید این مسیر طی شود.

## بخش ۴ — ناهماهنگی‌های سیستماتیک

### 🟡 پیام خطا در delete: سه سطح مختلف

| سطح | مثال | فایل |
|---|---|---|
| **خوب** — پیام اختصاصی | برگه انبار، شعبه، دسته | vouchers, branches, categories |
| **متوسط** — پیام slice error | `branchesError ?? 'خطا در حذف شعبه'` | BranchesPane, CategoriesPane |
| **بد** — `catch { showToast('خطا در حذف', 'danger') }` | قلم انبار، دستور پخت | inventory/page.tsx:217,480 |

---

### 🟡 soft-delete بدون صفحه‌ی آرشیو

موجودیت‌هایی که soft-delete دارند ولی هیچ راهی برای مشاهده یا بازگردانی ندارند:
- قلم انبار، دستور پخت، پرسنل، صندوق، طرف‌حساب، کوپن، مشتری، میز

این یعنی عملاً «حذف دائمی از دید کاربر» — با همه‌ی معایب hard delete ولی بدون هیچ مزیتی.

---

### 🟡 approve/reject workflow ناهماهنگ

| موجودیت | دارد؟ |
|---|---|
| تراکنش | ✅ pending → approved/rejected |
| برگه انبار | ✅ pending → approved/rejected |
| درخواست استخدام | ❌ فقط status string، بدون workflow |
| رزرو | ❌ status string، بدون workflow |
| اجرای حقوق | ✅ draft → calculated → approved → posted |

درخواست استخدام یک `status` دارد (pending/reviewing/...) ولی هیچ API route برای تغییر status آن وجود ندارد — فقط از طریق PATCH مستقیم.

---

### 🟢 قابل‌قبول — denormalization عمدی

`branch_name` و `category_name` در تراکنش‌ها کپی می‌شوند تا با حذف شعبه یا دسته، تاریخچه سالم بماند. این یک تصمیم درست است.

---

### 🟢 قابل‌قبول — FK های `restrict` جلوی حذف تصادفی را می‌گیرند

شعبه‌ای که تراکنش دارد حذف نمی‌شود. دسته‌ای که تراکنش دارد حذف نمی‌شود. این رفتار درست است. ولی پیام خطا باید واضح‌تر باشد («۴۵ تراکنش به این شعبه متصل است» بهتر از «این شعبه دارای تراکنش است» است).
